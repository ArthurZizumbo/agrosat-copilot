### 2.1 Decisión técnica — orquestación dual (script + asset Dagster)

El CA exige "Script `scripts/build_parcel_features.py` con asset Dagster". Trade-off:

- **Opción A (recomendada)**: el script CLI y el asset Dagster comparten el **mismo punto de entrada** `ml/features/fusion.py::build_fused_features`. El script es un wrapper Typer fino; el asset es un wrapper `@asset` fino. Una sola fuente de verdad para los joins, los stats y la salida parquet.
- Opción B: duplicar lógica en script + asset. Rechazada — viola DRY (regla §7 root CLAUDE.md).

**Decisión: Opción A.** El módulo `ml/features/fusion.py` expone funciones puras; script y asset son adaptadores I/O.

### 2.2 Decisión técnica — Sentinel-1 ingest mínimo en esta US

El repo no tiene `sample_s1_roi`. Crear el helper completo es scope de una US-007.x dedicada. **Mitigación**: esta US añade un sampler mínimo `sample_s1_roi_for_parcels(parcels_gdf, year)` con preset operativo (GRD IW VV+VH, asc+desc fusionado, despeckle Lee 7×7, sigma0 en dB) suficiente para los 10 cols de stats. Documentado como deuda técnica a refactorizar si una US futura amplía el sampler (ej. polarimetric textures).

### 2.3 Decisión técnica — granularidad temporal de stats S1 vs índices ópticos

S2 ópticos: stats anuales sobre ~30 timesteps. S1 backscatter: stats anuales sobre ~60 timesteps (revisita 6 días). Decisión: **mismo set de stats `{mean, std, p25, p50, p95}` para uniformidad**; descomposición FFT y fenología NO se aplican a S1 en esta US (las stats agregadas capturan suficiente discriminabilidad para baselines RF/XGB). Si una US futura entrega arquitecturas temporales explícitas (TSViT), las series temporales raw seguirán disponibles en cache parquet de `sample_s1_roi`.

### 2.4 Decisión técnica — h3 vs voronoi para spatial K-fold

Alternativas evaluadas:

- **h3 res 5 (~252 km²) + KMeans sobre centroides hex** (recomendada): librería madura (`h3-py`), determinista, fácil de visualizar, buffer 1 km inter-fold trivial con `h3.k_ring(1)` excluyendo bordes.
- Voronoi sobre puntos seed aleatorios: requiere `scipy.spatial.Voronoi`, más complejo de auditar.
- Tessellation rectangular fija: simple pero produce folds desbalanceados cuando las parcelas están concentradas en una región.

**Decisión: h3 res 5**. Añadir `h3 = "^4.1.2"` al grupo `geo` de `pyproject.toml` (decisión no-autónoma de stack — coordinar con Arthur antes del coding). Fallback documentado: si h3 no se aprueba, usar `geopandas.GeoSeries.bounds` + grid rectangular 50×50 km con merge de celdas vacías.

### 2.5 Decisión técnica — scaler global vs por-fold

CA dice "z-score global". Interpretación:

- **Global = un solo scaler ajustado sobre el split train del Fold-0** (recomendada). Simplifica downstream; los baselines de US-019 cargan un único scaler. Trade-off: ligero leakage si Fold-0 no es representativo, pero CV reporta std entre folds.
- Por-fold: 5 scalers (uno por fold train). Más estricto pero complica el pipeline de inferencia.

**Decisión**: scaler global del Fold-0 (`artifacts/scaler_v1.pkl`) **+** opcionalmente persistir 5 scalers por fold (`artifacts/scaler_v1_fold_{0..4}.pkl`) si US-018 los pide. Esta US entrega solo el global por simplicidad.

### 2.6 Decisión técnica — JSONB upstream vs recomputo on-the-fly

Los stats de NDVI ya viven en `features_parcels.ndvi_stats` JSONB (US-015) pero solo para NDVI. Para los otros 16 índices, US-015 NO los persistió (sería 17×153 cols escalares). Trade-off:

- **Opción A**: recomputar los 16 índices restantes on-the-fly con `compute_index_timeseries` (cache Redis ya implementado en US-014 cubre el costo repetido). Recomendada.
- Opción B: ampliar `features_parcels` con más JSONB para los 16 restantes (deuda upstream + migración + persistencia).

**Decisión: Opción A**. La fusión es offline batch; el cache Redis de US-014 reduce el costo a ~1× por (parcel, year, index). Para los 85 stats, se subset a 5 stats (no los 9 de US-015) por economía. El subset declarado: `{mean, std, p25, p50, p95}` (cubre tendencia central + dispersión + colas).

from typing import Literal

import polars as pl
import geopandas as gpd

FUSION_STATS: tuple[str, ...] = ("mean", "std", "p25", "p50", "p95")
BLOCK_NAMES = (

    "era5_monthly", "geometry", "farslip",

    year: int,
    *,
    blocks: tuple[str, ...] = BLOCK_NAMES,
    include_farslip: bool = False,
    farslip_path: str | None = None,
    stats: tuple[str, ...] = FUSION_STATS,
    lazy: bool = True,
) -> pl.DataFrame:
    """Construye el vector de features fusionados por (parcel_id, year).

    Args:
        parcels: GeoDataFrame con columnas `parcel_id`, `year`, `geom`
            (POLYGON EPSG:4326), opcionalmente `crop_class`, `region`.

### 4.2 `ml/features/spatial_split.py`

```python
from dataclasses import dataclass

import geopandas as gpd

@dataclass(frozen=True)
class FoldAssignment:
    fold_id: int
    train_ids: tuple[int, ...]
    val_ids: tuple[int, ...]
    test_ids: tuple[int, ...]


def build_spatial_kfold(
    parcels: gpd.GeoDataFrame,
    *,
    k: int = 5,
    h3_res: int = 5,
    buffer_km: float = 1.0,
    val_fraction: float = 0.2,
    random_state: int = 42,
) -> list[FoldAssignment]:
    """Construye K folds espaciales con tessellation H3 + KMeans sobre centroides hex.

    Args:
        parcels: GeoDataFrame con `parcel_id`, `geom` EPSG:4326.
        k: número de folds (default 5).
        h3_res: resolución H3 (default 5 ≈ 252 km²).
        buffer_km: distancia mínima entre parcelas de folds vecinos.
        val_fraction: fracción del train interno usada como val por fold.
        random_state: seed determinista para KMeans y val split.

    Returns:
        Lista de K FoldAssignment. Test fold = 1/K parcelas; val ⊂ train.
        Garantiza: (a) sin parcel_id en >1 fold, (b) distancia ≥ buffer_km
        entre folds vecinos, (c) class balance ratio max/min < 1.5 si
        `crop_class` está en parcels.
    """
```
