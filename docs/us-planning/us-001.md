## 0. Resumen ejecutivo

El monorepo `agro_sat_copilot` ya existe con la estructura completa al ~90 % del DoD. La US-001 originalmente pedía generar un repo externo `gh:agrosatcopilot/cookiecutter-agrosat` que produzca el árbol vía `cookiecutter`. **Reinterpretación pragmática aprobada** (ver §11 Decisiones del plan):

1. **No publicar un repo cookiecutter externo**. El proyecto es académico (10 sem, 3 devs), no SaaS multi-cliente: un cookiecutter externo agrega mantenimiento sin valor — el monorepo actual ya cumple la función de "single source of truth de la estructura".
2. **No prompts interactivos** (`project_name`, `gcp_project_id`, `azure_subscription_id`, `region`, `db_name`, `team_lead_email`). Esos valores viven en `.env.local` (dev) y `terraform.tfvars` (infra). Pedirlos al generar dispersaría la configuración en tres sitios.
3. **Un único entorno `dev`** (no `staging/prod`). Coherente con presupuesto académico, scale-to-zero, y calendario inamovible (presentación 21-jun-2026).
4. **Sí entregar**: completar los huecos reales del DoD (LICENSE MIT, Dockerfiles multi-stage, `cloudbuild.yaml`, módulos Terraform base GCP+Azure con valores ya hardcodeados al proyecto, workflow CI mínimo en `.github/workflows/`, scripts faltantes, `ml/configs/l4_spot.yaml`).

La US se cierra cumpliendo los **criterios de aceptación reinterpretados** que firme el equipo (§1) y dejando documentado en `docs/decisions/ADR-001-no-cookiecutter-externo.md` el motivo de la desviación respecto al plan v6.

### 2.2 Flujo onboarding (objetivo principal de la US)

```
git clone <repo>
   │
   ├─► cp .env.example .env.local  ← editar GCP_PROJECT_ID, CLERK_*, HF_TOKEN
   │
   ├─► make dev                    ← levanta 8 contenedores (~90 s)
   │
   ├─► make db-migrate             ← dbmate up (aplica initial_schema)
   │
   └─► open http://localhost:3000  ← Nuxt UI lista; chat conecta a :8000
```

Diferencia respecto a un cookiecutter clásico: el clon **ya** trae el árbol y los archivos generados; el setup interactivo se hace una vez en `.env.local` (variables que el dev real necesita) y otra vez en `terraform.tfvars` (cuando se quiera desplegar a GCP).

## 11. Decisiones del plan (resumen ADRs)

**ADR-001 — No publicar cookiecutter externo**

- *Contexto*: Plan v6 §EPIC 0 pide `cookiecutter gh:agrosatcopilot/cookiecutter-agrosat`.
- *Decisión*: Mantener el monorepo como artefacto canónico; saltar publicación del repo cookiecutter externo.
- *Consecuencias positivas*: −2 SP de mantenimiento, +simplicidad onboarding (un solo `git clone` + `make dev`), evita doble fuente de verdad.
- *Consecuencias negativas*: pierde generalización a "spinear un proyecto similar". Aceptable: el proyecto es académico de 10 sem, no producto multi-tenant.
- *Status*: Propuesta — requiere visto bueno equipo + Dr. Camacho.

**ADR-002 — Un solo entorno `dev` durante el curso**

- *Contexto*: Plan v6 menciona `dev`, `staging`, `prod`. Calendario sólo llega a presentación 21-jun.
- *Decisión*: Mantener `infrastructure/terraform/environments/dev/`. Vaciar `staging/` y `prod/` con README "fuera de scope académico".
- *Consecuencias positivas*: −presupuesto cloud (1 Cloud SQL en vez de 3), −complejidad CI/CD.
- *Consecuencias negativas*: si se decidiera publicar SaaS post-curso, requeriría reintroducir entornos. Aceptable: ese es trabajo del Paper Track o post-curso.
- *Status*: Propuesta — requiere visto bueno equipo.

**Sin prompts interactivos en setup**: `project_name` se infiere del nombre del directorio; `gcp_project_id` vive en `terraform.tfvars` y `.env.local`; `azure_subscription_id` ídem; `region` hardcoded `europe-west1` (proximidad Italia, justificado en módulo Terraform); `db_name` hardcoded `agrosat` (consistente con `docker-compose.yml` actual); `team_lead_email` en `pyproject.toml` y `CODEOWNERS`.
