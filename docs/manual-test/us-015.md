# Manual Test — US-015 · Features temporales agregados por parcela

**Epic**: E3 · **Avance**: A2 · **Sprint**: S4 · **Estado**: ready-to-close

> US-015 es 100 % ML + DB (sin UI, sin chat, sin LLM). La mayoría de criterios se
> validan automáticamente vía `pytest`. Esta guía cubre los flujos que requieren
> juicio humano: ejecutar el extractor end-to-end, inspeccionar visualmente la
> salida del DataFrame y validar las migraciones en una Postgres real.

---

## Pre-requisitos

- Repo clonado y `poetry install` ejecutado en `ml/` (y opcionalmente `backend/`).
- Python 3.12.x + Poetry ≥ 1.7.
- Para los flujos DB: Docker Desktop corriendo + `psycopg[binary]` instalado
  (`cd ml && poetry add --group dev psycopg[binary]` — pendiente de aprobación
  de equipo, ver handoff §"Desvíos").
- Fixture demo presente: `data/test_fixtures/parcel_demo_ts.nc` (≈ 4.9 KB).

---

## Flujo 1 — Extracción end-to-end sobre fixture demo (10 min)

**Objetivo**: validar AC-1..AC-4 (stats + FFT + fenología + slopes) con datos
sintéticos controlados y revisar la salida humanamente.

| Paso | Comando o acción | Resultado esperado |
|------|------------------|---------------------|
| 1.1 | `make features-extract-demo` desde la raíz | Sale `features extracted shape=(1, 187)` con los primeros 8 nombres de columna (`parcel_id, year, NDVI_mean, NDVI_std, NDVI_min, NDVI_max, NDVI_p05, NDVI_p25`) |
| 1.2 | Lanzar Python REPL: `poetry run python` desde la raíz | Prompt interactivo |
| 1.3 | Pegar: `import xarray as xr; from ml.features import extract_temporal_features; ds = xr.open_dataset('data/test_fixtures/parcel_demo_ts.nc'); da = ds['parcel_indices']; da = da.assign_coords(band=[b.decode() for b in da.coords['band'].values]); df = extract_temporal_features(da)` | DataFrame Polars creado sin warnings ni excepciones |
| 1.4 | `df.shape` | `(1, 187)` |
| 1.5 | `df.select('peak_doy', 'peak_value', 'sog_doy', 'senescence_doy').to_dicts()` | `peak_doy ∈ [150, 210]` (gaussiana centrada en DOY 180 con ruido σ=0.02 + grilla cada 12 días → el pico observado en el fixture concreto es DOY 169; el test unitario `test_phenology_synthetic_gaussian` valida [178, 182] sobre curva sin ruido), `peak_value ≈ 0.85 ± 0.05`, `sog_doy` y `senescence_doy` plausibles (típicamente ~120 y ~240 respectivamente) |
| 1.6 | `df.select(pl.col('^NDVI_fft_amp_.*$')).row(0)` | 4 valores; `NDVI_fft_amp_0` ≈ media del ciclo (~0.3-0.5), armónico anual `_1` debe ser el mayor |
| 1.7 | `df.select('ndvi_slope_pre_peak', 'ndvi_slope_post_peak').row(0)` | Pre > 0 (subida), post < 0 (bajada). Magnitudes similares (gaussiana ≈ simétrica) |

**Criterios de aceptación cubiertos**: AC-1, AC-2, AC-3, AC-4, AC-7
(determinismo: re-ejecutar 1.3 dos veces da DataFrames idénticos).

## Flujo 2 — Migraciones reversibles en Postgres real (15 min)

**Objetivo**: validar AC-5 (schema, UNIQUE, FK CASCADE, índices) con un Postgres
con PostGIS + pgvector reales (los tests `tests/db/test_migrations_us015.py`
están skip clean por falta de `psycopg2`/Docker en dev local).

| Paso | Comando o acción | Resultado esperado |
|------|------------------|---------------------|
| 2.1 | `make dev` (levanta docker-compose con Postgres 15 + PostGIS + pgvector) | Servicios up, puerto 5432 abierto |
| 2.2 | `make db-migrate` (aplica todas las migraciones, incl. las dos de US-015) | Salida `Applied: 20260516210000_create_parcels.sql` y `20260516210100_create_features_parcels.sql` sin errores |
| 2.3 | `make db-shell` y ejecutar `\d parcels` | Tabla con 10 columnas: `id BIGSERIAL`, `session_id UUID`, `aoi_id BIGINT`, `geom geometry(Polygon,4326) NOT NULL`, `crop_class TEXT`, `confidence REAL`, `area_ha REAL`, `year SMALLINT NOT NULL`, `created_at`, `updated_at`. FKs visibles a `chat_sessions(id)` y `aois(id)` |
| 2.4 | `\d features_parcels` | Tabla con 14 columnas incluyendo `parcel_id BIGINT NOT NULL`, `year SMALLINT NOT NULL`, `alphaearth_embedding vector(64)`, `ndvi_stats jsonb NOT NULL DEFAULT '{}'::jsonb`, `phenology jsonb NOT NULL DEFAULT '{}'::jsonb`, 8 escalares fenológicas. Constraint `features_parcels_parcel_year_uniq UNIQUE (parcel_id, year)`. Índices BTREE en `parcel_id` y `year` |
| 2.5 | `\di parcels_geom_idx` | Index `parcels_geom_idx` USING GIST sobre `geom` |
| 2.6 | Probar UNIQUE: `INSERT INTO chat_sessions (user_id) VALUES ('test') RETURNING id;` (guardar `id`), `INSERT INTO parcels (session_id, geom, year) VALUES ('<id>', ST_GeomFromText('POLYGON((0 0,1 0,1 1,0 1,0 0))', 4326), 2024) RETURNING id;` (guardar `parcel_id`), `INSERT INTO features_parcels (parcel_id, year) VALUES (<parcel_id>, 2024);` luego `INSERT INTO features_parcels (parcel_id, year) VALUES (<parcel_id>, 2024);` | El segundo INSERT lanza `ERROR: duplicate key value violates unique constraint "features_parcels_parcel_year_uniq"` |
| 2.7 | Probar CASCADE: `DELETE FROM parcels WHERE id = <parcel_id>;` luego `SELECT * FROM features_parcels WHERE parcel_id = <parcel_id>;` | Cero filas (la fila de features se borró en cascada) |
| 2.8 | Round-trip: `make db-rollback` (twice si dbmate baja una a una), `make db-migrate` | Schema reaplicado idéntico. `\d features_parcels` muestra lo mismo que en 2.4 |
| 2.9 | `make stop` | Servicios bajados |

**Criterios de aceptación cubiertos**: AC-5 (todos los sub-checks).

## Flujo 3 — UPSERT y modos de conflicto desde Python (10 min)

**Objetivo**: validar `load_features_parcels` en sus 3 modos (`update`, `skip`,
`raise`) contra una DB real. Equivalente Python de los tests
`test_persist_features.py` (que están skip clean por falta de psycopg).

| Paso | Comando o acción | Resultado esperado |
|------|------------------|---------------------|
| 3.1 | Repetir flujo 2 pasos 2.1-2.4 | Schema listo |
| 3.2 | Crear una parcela demo en Python: usar engine `sqlalchemy.create_engine(os.environ['DATABASE_URL'])` + INSERT como en 2.6 hasta obtener `parcel_id` | `parcel_id` válido en variable |
| 3.3 | Crear DataFrame trivial: `import polars as pl; df = pl.DataFrame({'parcel_id': [<parcel_id>], 'year': [2024], 'sog_doy': [120], 'peak_doy': [180], 'peak_value': [0.85], 'senescence_doy': [240], 'ndvi_auc': [50.0], 'ndvi_slope_pre_peak': [0.005], 'ndvi_slope_post_peak': [-0.005], 'maturity_duration_days': [40]})` | DF Polars 1×10 |
| 3.4 | `from ml.features.persist_features import load_features_parcels; load_features_parcels(df, engine)` | Retorna `1`. `SELECT * FROM features_parcels WHERE parcel_id = <parcel_id>` muestra la fila |
| 3.5 | Modificar `df` (cambiar `peak_value` a `0.90`) y re-llamar `load_features_parcels(df, engine)` (modo default `update`) | Retorna `1`. `SELECT peak_value FROM features_parcels WHERE parcel_id = <parcel_id>` ahora muestra `0.90`. `updated_at` cambió |
| 3.6 | `load_features_parcels(df, engine, on_conflict='skip')` con el mismo df | Retorna `0`. La fila NO se modifica |
| 3.7 | `load_features_parcels(df, engine, on_conflict='raise')` con el mismo df | Lanza `sqlalchemy.exc.IntegrityError` |
| 3.8 | `load_features_parcels(pl.DataFrame({'parcel_id': [], 'year': []}), engine)` | Retorna `0` sin errores (frame vacío) |

**Criterios cubiertos**: AC-5 (UPSERT), AC-7 (reproducibilidad operativa).
