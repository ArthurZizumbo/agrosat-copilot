## Bugs corregidos post-handoff (2026-05-13, revisión Arthur)

### BUG-1 · `init_ee` disparaba `ee.Authenticate()` interactivo bloqueando papermill/CI

**Síntoma**: en `notebooks/02b_eda_alphaearth.ipynb` celda 3, cualquier fallo
de `ee.Initialize` (proyecto sin EE habilitado, `GOOGLE_APPLICATION_CREDENTIALS`
roto, etc.) caía en un `try/except` que invocaba `ee.Authenticate()` — el cual
abre browser y espera input en stdin. En papermill o `nbconvert --execute` el
proceso quedaba colgado indefinidamente, no degradaba como decía el snapshot.

**Root cause**: `ml/ingest/gee_sampler.py:init_ee` tenía hardcoded el fallback
a auth interactivo, sin opción de propagar el error.

**Fix**:

- `init_ee` ahora acepta `interactive_auth: bool = False`. Default propaga
  excepción; solo dispara `ee.Authenticate()` si el caller lo pide explícito.
- `init_ee` también valida que `service_account_json` exista en disco antes
  de pasárselo a `ee.ServiceAccountCredentials` (evita fallo si el path del
  `.env.local` es placeholder).
- Tests actualizados: `test_init_ee_propagates_error_by_default` reemplaza
  `test_init_ee_falls_back_to_authenticate_on_initialize_error`. Nuevo test
  `test_init_ee_skips_service_account_if_file_missing`. 23/23 pasan.

### BUG-2 · Notebook no leía `GEE_PROJECT_ID` de `.env.local`

**Síntoma**: celda 3 llamaba `init_ee()` sin argumentos → `ee.Initialize`
caía al ADC del gcloud (`nupciapp`) y fallaba con "Earth Engine API has not
been used in project nupciapp".

**Fix** (notebook celda 3):

- Loader inline `_load_env_local(REPO)` que parsea `.env.local` sin
  dependencia python-dotenv y popula `os.environ`.
- Limpia `GOOGLE_APPLICATION_CREDENTIALS` si apunta a archivo inexistente
  (es el caso del template `.env.local`: `./.env/gcp-service-account.json`
  no existe en dev local).
- Pasa `project=GEE_PROJECT_ID` y `service_account_json` (si el archivo
  existe) explícitos a `init_ee`.

### BUG-5 · `pastis_pixel_labels` retornaba `class_name="unknown"` para todos los pixels

**Sintoma**: en la primera ejecucion del notebook (Jupyter desde otro CWD)
los 96,223 pixels PASTIS-R cargados quedaban con `class_name="unknown"` aunque
el `class_id` fuera valido (1..18). t-SNE/UMAP/RF se ejecutaban pero con una
sola clase aparente, destruyendo la senal cross-region.

**Root cause**: `_CLASS_MAP_PATH = Path("data/reference/pastis_class_mapping.json")`
es **relativo al CWD**. Si Jupyter / papermill arranca con CWD distinto al
repo root, `_CLASS_MAP_PATH.exists()` retorna False -> `_load_class_mapping`
cae al fallback minimalista `{0: "Background", 19: "Void label"}` -> todos los
class_ids 1..18 caen al `default="unknown"` del `.replace_strict`.

**Fix** (`ml/ingest/pastis_loader.py:41-43`):

```python
_REPO_ROOT = Path(__file__).resolve().parents[2]
_DEFAULT_ROOT = _REPO_ROOT / "data" / "PASTIS-R"
_CLASS_MAP_PATH = _REPO_ROOT / "data" / "reference" / "pastis_class_mapping.json"
```

Ahora las rutas son absolutas y ancladas al repo, independientes del CWD.

### BUG-6 · `tsne_2d` / `umap_2d` / `rf_feature_importance` reventaban con NaN

**Sintoma**: en cell 18 del notebook (t-SNE Francia), `ValueError: Input X
contains NaN. TSNE does not accept missing values encoded as NaN natively.`

**Diagnostico inicial incorrecto** (auditado tras pregunta del revisor):
en la primera ronda atribui el NaN a "pixels PASTIS fuera del footprint
AlphaEarth 2019" y metí `dropna` como fix. **Esto era un parche cosmético
que enmascaraba un bug real (BUG-7 abajo)**.

**Fix aplicado en la primera ronda** (cinturon-y-tirantes, defensa razonable):

- `tsne_2d`: filtra `valid_idx = np.flatnonzero(~np.isnan(arr).any(axis=1))`
  antes del subsample. Los `idx` retornados son contra el df original, asi
  que `df[col].to_numpy()[idx]` sigue alineado.
- `umap_2d`: mismo patron. **Breaking change minimo**: ahora retorna tupla
  `(X_2d, idx)` en vez de solo `X_2d`.
- `rf_feature_importance`: filtra filas con NaN en X **y** labels nulos.
- Notebook cells 9 y 18: ahora desempaquetan `X_umap, idx_umap = umap_2d(...)`.
- Notebook cells 10 y 19 (RF Italia/Francia): guarda `if not rf_X.is_empty()`
  antes de `rf_X['oob_score'][0]`.

**Mejora post-auditoria**: cada `dropna` emite `_log.warning("*_dropped_nan_rows",
n_dropped, n_total, pct)` via `structlog` para hacer visible cualquier futura
contaminacion de NaN (no oculta problemas downstream). Tras BUG-7 fix, sobre
data real estos warnings no se emiten (0% NaN — verificado).

Tests: +3 nuevos (`test_umap_2d_drops_nan_rows`, `test_tsne_2d_drops_nan_rows`,
`test_rf_feature_importance_drops_nan_rows`). `test_umap_2d_shape_or_empty`
actualizado al nuevo contrato.

### BUG-7 (RAIZ REAL) · `sample_alphaearth_*().first()` retornaba data vacia

**Sintoma**: `sample_alphaearth_at_coords` retornaba **0 filas** o
embeddings con NaN para coords PASTIS que SI tenian cobertura AlphaEarth.
La coleccion `GOOGLE/SATELLITE_EMBEDDING/V1/ANNUAL` tiene ~10,690 imagenes
por anio en Europa (tiles disjuntos UTM).

**Root cause**: tanto `sample_alphaearth_at_coords` (linea 392) como
`sample_alphaearth_roi` (linea 298) usaban `collection.first().select(...)`.
`first()` devuelve **un tile arbitrario** con footprint local (~110 km²).
Cualquier punto fuera de ese tile cae a null. Para PASTIS-R en Bretaña los
puntos NO caian en el tile arbitrario seleccionado por `first()` -> 0 features
con A00 no-null. **Esto invalidaba la Seccion 2 entera del EDA.**

Diagnostico empirico (`scripts/_diag_alphaearth_call.py`, temporal):

```
Test 3: hay 1 imagen AlphaEarth 2019 cubriendo (-1.09, 49.43): SI
Test 4: con mosaic() en 5 puntos PASTIS reales: 5/5 con A00 valido
Test 5: con first() (codigo actual): 0/5 con A00 no-null
```

**Fix** (`ml/ingest/gee_sampler.py:298, 392`):

```python
# antes
image = collection.first().select(band_names)
# despues
image = collection.mosaic().select(band_names)
```

`mosaic()` une todos los tiles del anio en un raster continuo. AlphaEarth
publica 1 embedding por (pixel, anio), asi que `mosaic()` es deterministico
(no hay ambiguedad por overlap). Validado con 600 pixeles PASTIS reales
(patches 10000-10002, Bretaña):

```
Filas totales:               600
Filas con NaN (any):           0  (0.00%)
NaN por clase (8 clases):     0.0% spread
NaN por patch (3 patches):    0.0% spread
RF OOB score Francia (real):  0.8028   <-- senal cientifica valida
```

Tests: mocks `_build_alphaearth_reduce_regions_ee` (test_gee_sampler.py:278)
y `_fake_alphaearth_ee` (test_embeddings.py:346) actualizados a cablear
`fake_collection.mosaic` en vez de `fake_collection.first`.

### BUG-8 · ROOT/REPO path hardcoded en notebooks (cwd-dependent)

**Sintoma**: al ejecutar el notebook desde `notebooks/eda/`, `Patches totales:
0` y "PASTIS metadata no disponible - modo degradado con sintetico" aunque
`data/PASTIS-R/metadata.geojson` SI existe en el repo.

**Root cause**: los 3 notebooks tenian heuristicas fragiles para resolver el
repo root, todas validas SOLO en una location especifica:

| Notebook | Linea problematica | Falla cuando |
|----------|--------------------|--------------|
| `02a_eda_sentinel2.ipynb` | `ROOT = Path.cwd().parent if Path.cwd().name == "notebooks" else Path.cwd()` | cwd = `notebooks/eda/` (notebook ya no esta en `notebooks/` directo) |
| `02b_eda_alphaearth.ipynb` | `REPO = Path.cwd() if Path.cwd().name == "agro_sat_copilot" else Path.cwd().parent` | cwd = `notebooks/eda/` (sube solo 1 nivel) |
| `02c_eda_pastis.ipynb` (Isaac) | `DATA_PATH = Path('../data/PASTIS-R')` (hardcoded relative) | cwd != `notebooks/eda/` |

Consecuencias:
- `02a`: `ROOT = notebooks/eda/` -> `PASTIS = notebooks/eda/data/PASTIS-R/`
  (no existe) -> cae a modo sintetico.
- `02b`: igual cascada via `REPO`, ademas creo `notebooks/eda/data/cache/` y
  `notebooks/eda/paper/figures/` artefactos.
- `02c`: igual via `DATA_PATH`.
- `_load_env_local(REPO)` con REPO mal apuntado no encuentra `.env.local` ->
  `GEE_PROJECT_ID=None` -> EE arranca sin project ID valido.

**Fix**: helper `_find_repo_root(start)` que busca `pyproject.toml` subiendo
niveles desde `start`. Aplicado a los 3 notebooks:

```python
def _find_repo_root(start: Path) -> Path:
    for candidate in (start.resolve(), *start.resolve().parents):
        if (candidate / "pyproject.toml").is_file():
            return candidate
    return start.resolve()


REPO = _find_repo_root(Path.cwd())
```

Funciona desde `notebooks/`, `notebooks/eda/`, `scripts/`, o cualquier
subdirectorio del repo. Verificado:

```

PASTIS exists: True
metadata.geojson exists: True
```

### BUG-11 · `sample_dynamic_world_at` y `sample_alphaearth_at_coords` no batchean → respuesta vacía con >1000 coords

**Síntoma**: con 6,000 coords, `display(df_italia.group_by("dw_class_name"))`
retornaba `dw_class_name=null, n=6000`. El join con DW devolvía null para
TODAS las filas pero el sampler no levantaba excepción.

**Verificación experimental (no parche)**:
- 10 coords: 6 clases distintas correctamente. ✓
- 100 coords: 6 clases distintas correctamente. ✓
- 6,000 coords (sin batching): payload vacío, todos null. ✗

**Root cause**: GEE `reduceRegions` con 6,000 features en una sola request
excede el límite del compute graph (5 min server-side), retorna respuesta
parcial/vacía pero **sin lanzar excepción**. El código procesaba
`info["features"] = []` → DataFrame vacío con esquema → join con cardinalidad
del padre produce todos null en la columna de la derecha.

**Fix en `ml/ingest/gee_sampler.py`**:

- `sample_dynamic_world_at`: nuevo parámetro `batch_size: int = 500`. Procesa
  coords en lotes. Si un batch falla, marca sus puntos con `dw_class_id=-1`,
  `dw_class_name="unknown"` para preservar cardinalidad downstream.
- `sample_alphaearth_at_coords`: mismo tratamiento (cell-11 estabilidad
  temporal con 4 años × 500 puntos y cell-16 con miles de puntos PASTIS lo
  necesitaban igual).
- También corregido orden de fallback `props.get("first", props.get("label"))`
  porque `reduceRegions + Reducer.first()` renombra la banda a `"first"`, no
  preserva el nombre `"label"` original.

**Tests**: nuevo `test_sample_dynamic_world_at_batches_large_coords` (1,200
coords → 3 batches × 500). `test_sample_dynamic_world_at_degraded_on_exception`
actualizado al nuevo contrato (marca `unknown` en vez de DataFrame vacío).

### BUG-13 · `fetch_s2_ndvi_rgb_for_parcel` retornaba imagen 1×1 (un solo pixel)

**Síntoma**: con el fix de BUG-12 aplicado, el RGB seguía pareciendo un cuadro
uniforme. Print del shape reveló `RGB shape=(1, 1, 3) NDVI shape=(1, 1)
std_per_band=[0.0, 0.0, 0.0]`.

**Verificación experimental (no parche)**:
- Imagen original Sentinel-2: `EPSG:32632, scale=10m` ✓
- Después de `.median()`: **`EPSG:4326, scale=1 grado/pixel`** ✗
- Bbox de 0.01° en proyección de 1°/pixel = **menos de 1 pixel**.
- Reproyectando con `.reproject(crs=ref_proj, scale=10)`: `(114, 81)` ✓.

**Root cause**: comportamiento documentado de GEE. `ImageCollection.median()`
colapsa la proyección a la default (WGS84, 1 grado por pixel) si no se
especifica explícitamente. `sampleRectangle` sobre esa imagen con un bbox
chiquito en grados decimales colapsa a 1×1.

**Fix en `ml/ingest/gee_sampler.py:fetch_s2_ndvi_rgb_for_parcel`**:

```python
ref_proj = collection.first().select("B4").projection()       # UTM nativo S2
median = collection.median().reproject(crs=ref_proj, scale=scale)
sample = median.sampleRectangle(region=parcel_geom, defaultValue=0)
```

Extrae la proyección UTM de la primera imagen S2 y reproyecta el median a
10 m antes de `sampleRectangle`. Ahora retorna ~110×80 pixels reales.

**Tests**: mock `_build_fetch_s2_ee` actualizado para incluir la chain
`.first().select().projection()` + `.median().reproject().sampleRectangle()`.
24/24 pasan.

### BUG-14 · `fetch_s2_ndvi_rgb_for_parcel` stretch RGB global colapsaba canales

**Síntoma**: con BUG-13 fixed, RGB seguía viéndose verde-amarillo uniforme.

**Root cause**: el código aplicaba `np.percentile(rgb, [2.0, 98.0])`
globalmente sobre el stack 3D ya combinado. En vegetación, B4 ~3000, B3 ~2000,
B2 ~1500 reflectancia escalada → un percentil global elige un único rango
para las 3 bandas → el azul queda casi en 0, el rojo se satura → todo el RGB
cae en un tono uniforme.

**Fix**: stretch percentil 2-98 **por banda independiente** antes del stack.
Cada canal usa su propio rango dinámico.

### BUG-15 · Kernel Jupyter cacheaba `gee_sampler.py` editado

**Síntoma**: tras aplicar fix BUG-13 en disco, re-ejecutar cell-12 seguía

Documentado como regla obligatoria en `notebooks/CLAUDE.md` §"Hot-reload de
módulos `ml/*` en kernel Jupyter".
