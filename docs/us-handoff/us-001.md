## Hardening de seguridad вЂ” 2026-05-12 (tercera pasada)

Tercera revision focalizada en hallazgos de seguridad fuera del diff original.
4 fixes aplicados:

### SEC-1 вЂ” CORS `allow_headers=["*"]` con `allow_credentials=True`

**Sev**: high В· **Archivo**: `backend/app/main.py:51`

`CORSMiddleware` combinaba `allow_credentials=True` con `allow_headers=["*"]`,
lo que permite que cualquier header arbitrario llegue con cookies.

**Fix**: whitelist explicita: `Authorization`, `Content-Type`, `Accept`,
`X-Request-ID`, `X-Session-ID`. Tambien agregado `expose_headers` para
`X-Request-ID` y `X-RateLimit-*` necesarios por el frontend.

### SEC-2 вЂ” pydantic Settings `extra="ignore"`

**Sev**: med В· **Archivo**: `backend/app/core/config.py`

Antes: typos en `.env.local` se ignoraban silenciosamente. Ahora `extra="forbid"`
aborta el arranque ante variables no declaradas.

**Fix**: declarado el `Settings` completo con las 50+ vars del `.env.example`
agrupadas por seccion (conexiones, cloud, GEE/CDSE, HF, LLM, MLflow, Clerk,
frontend, observabilidad, JWT, TF passthrough, DVC, host ports). Cualquier
variable nueva del `.env.example` que el backend deba leer debe agregarse
explicitamente aqui.

### SEC-3 вЂ” `DATABASE_URL` con default literal `agrosat:agrosat`

**Sev**: med В· **Archivo**: `backend/app/core/config.py`

Antes: si un deploy real arrancaba sin `.env.local` valido, conectaba con
credenciales por defecto del docker-compose.

**Fix**: `model_validator` que rechaza arranque cuando `env != "dev"` con:
- `database_url == _DEV_DATABASE_URL`
- `redis_url == _DEV_REDIS_URL` (a menos que haya Upstash configurado)
- `jwt_secret == _JWT_PLACEHOLDER`

Verificado con smoke test: en `env=prod` con defaults dev, lanza
`ValueError: DATABASE_URL es obligatorio cuando env='prod' ...`.

### SEC-4 вЂ” Azure Key Vault `Key Vault Administrator` al deployer

**Sev**: low В· **Archivo**: `infrastructure/terraform/modules/azure/main.tf:233`

CIS Azure Benchmark recomienda `Key Vault Secrets Officer` (no Administrator)
para deployers que solo gestionan secretos (no keys, certs, policies).

**Fix**: `role_definition_name` cambiado a `Key Vault Secrets Officer`. El
deployer ya no puede modificar policies del KV, solo CRUD de secretos.

### BUG-S6-1 вЂ” Backend Docker instala 417 paquetes (incluido stack CUDA)

**Sev**: high В· **Detectado por**: usuario al revisar el log del build observГі
`nvidia-cublas`, `nvidia-cusparse`, `torch+cu130`, `mlflow`, `dvc`, `transformers`
entre los paquetes instalados en el `dev` stage del backend.

**DiagnГіstico** (ver [`ADR-004`](../decisions/ADR-004-poetry-optional-groups-and-aiplatform-ml.md)):

1. `google-cloud-aiplatform[reasoning_engines]` estaba en grupo `main` в†’
   arrastraba transitivamente todo el SDK Vertex AI Agent Engine al backend.
2. Los grupos `ml`, `geo`, `dagster`, `paper` NO estaban marcados como
   `optional = true` в†’ `poetry install --with dev,test --no-root` (que usa
   `backend.Dockerfile`) los instalaba TODOS por default. El flag `--with`
   solo agrega opcionales; no excluye no-opcionales.

**Fix aplicado** (decisiГіn documentada en ADR-004):

```diff
# pyproject.toml
- [tool.poetry.dependencies]
- google-cloud-aiplatform = { extras = ["reasoning_engines"], version = "^1.152.0" }
+ # (movido al grupo ml вЂ” el agente ADK vive en ml/agent/, no en handlers FastAPI)

+ [tool.poetry.group.ml]
+ optional = true
  [tool.poetry.group.ml.dependencies]
+ google-cloud-aiplatform = { extras = ["reasoning_engines"], version = "^1.152.0" }
  # ...

+ [tool.poetry.group.geo]
+ optional = true
+ [tool.poetry.group.dagster]
+ optional = true
+ [tool.poetry.group.paper]
+ optional = true
```

**ValidaciГіn post-fix**:

|---------|-------|---------|-----------|

`make bootstrap` (que pasa `--with dev,test,ml,geo,dagster,paper` explГ­cito)

### BUG-S6-2 вЂ” `docker-compose.yml` context wrong para servicio `api`

**Sev**: high В· **Detectado por**: log del build con error
`COPY pyproject.toml poetry.lock* ./: "/pyproject.toml": not found`.

**DiagnГіstico**: `docker-compose.yml` declaraba:

```yaml

  build:
    context: ./backend          # в†ђ bug: pyproject.toml estГЎ en raГ­z, no en ./backend

```

**Fix aplicado**:

```diff

-     context: ./backend
-     dockerfile: ../infrastructure/docker/backend.Dockerfile

+     dockerfile: infrastructure/docker/backend.Dockerfile

```

**Frontend NO tiene este bug**: `frontend.Dockerfile` copia `package.json` y
`pnpm-lock.yaml` que SГЌ estГЎn en `./frontend/` (el contexto declarado).

## Quinta pasada вЂ” ModernizaciГіn de stack + lockfiles (2026-05-12)

Detonante: el manual-test (cuarta pasada) descubriГі `pnpm-lock.yaml` ausente y `poetry lock --check` con sintaxis Poetry 1.x.
