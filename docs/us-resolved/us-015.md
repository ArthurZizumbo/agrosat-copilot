## Resumen ejecutivo

US-015 entrega el **pipeline temporal por parcela**: dado un `xr.DataArray` con dims `(time, band)` y attrs `parcel_id`/`year`, [`extract_temporal_features`](../../ml/features/temporal_features.py) produce un `pl.DataFrame` Polars 1.x con **187 columnas** por `(parcel_id, year)`:

- **153 estadisticos descriptivos** = 9 stats (`mean, std, min, max, p05, p25, p50, p75, p95`) x 17 indices espectrales canonicos (los mismos de US-014).
- **24 columnas FFT** = 4 amplitudes + 4 fases (DC + 3 armonicos) x 3 indices clave (NDVI, NDWI, EVI), con pre-interpolacion lineal diaria para corregir la revisita irregular de Sentinel-2.
- **8 columnas fenologicas** sobre NDVI: `sog_doy, peak_doy, peak_value, senescence_doy, ndvi_auc, ndvi_slope_pre_peak, ndvi_slope_post_peak, maturity_duration_days`. Comportamiento *graceful*: si la curva NDVI nunca cruza el umbral SOG (default 0.3), las columnas se devuelven como `None` en lugar de excepcion.
- **2 columnas indice**: `parcel_id`, `year`.

La persistencia se hace con [`load_features_parcels`](../../ml/features/persist_features.py), que ejecuta UPSERT `ON CONFLICT (parcel_id, year)` con 3 modos (`update`/`skip`/`raise`) sobre la tabla `features_parcels`. Las **dos migraciones reversibles** ([`20260516210000_create_parcels.sql`](../../db/migrations/20260516210000_create_parcels.sql) + [`20260516210100_create_features_parcels.sql`](../../db/migrations/20260516210100_create_features_parcels.sql)) crean el schema completo segun plan v6 §4.3: `parcels` mínima como precondicion del FK + `features_parcels` con UNIQUE `(parcel_id, year)`, FK CASCADE a `parcels`, columna `alphaearth_embedding VECTOR(64)` reservada para fusion multisensor (US-016), y JSONB para los bloques `ndvi_stats` (153 stats) y `phenology` (24 FFT) que serian impracticos como columnas escalares.

La US es consumida aguas abajo por **US-016 fusion multisensor** (combina `features_parcels` con embeddings AlphaEarth), **US-017 seleccion/extraccion** (VIF, PCA, UMAP sobre el frame Polars) y **US-018 baseline RF/XGB** (consume features fusionados).

Durante el ciclo de QA se descubrieron y resolvieron **6 bugs** (B-1..B-5 + un fix de raiz para W-3 heredado del initial schema), todos en codigo de tests, infra dev local y un target Makefile — ninguno en el codigo de produccion del feature engineering. La US queda con **91 tests pass, 0 skip, 0 fail** y cobertura **93 % en `temporal_features.py`** (target plan 80 %).

### Multi-tenancy y `session_id`

US-015 **no expone endpoints** ni codigo de runtime que consulte `parcels` o `features_parcels`. La proteccion multi-tenant se hereda del schema:

- `parcels.session_id UUID REFERENCES chat_sessions(id) ON DELETE CASCADE` — clave de tenancy.
- `features_parcels.parcel_id BIGINT REFERENCES parcels(id) ON DELETE CASCADE` — heredada via JOIN con `parcels`.

**Pendiente para US futuras** (NO bloqueante para US-015): cuando US-018+ expongan `features_parcels` en `/parcels/*` o `/timeseries/*`, deben:

1. Filtrar `WHERE EXISTS (SELECT 1 FROM parcels p WHERE p.id = features_parcels.parcel_id AND p.session_id = :session_id)` en cada query.
2. Anadir tests cross-session que confirmen que un `session_id` no ve `features_parcels` de otro.
3. Cerrar **W-5** (gap heredado del initial schema): `ENABLE ROW LEVEL SECURITY` + `CREATE POLICY ... USING (session_id = current_setting('app.session_id')::uuid)` para `aois`, `parcels`, `features_parcels` (via subquery a parcels) y `agent_sessions`.
